## Doctor checks

`figma-mcp doctor` runs six parallel health checks. Each entry below maps to one check ID; clicking through from doctor's output lands here.

### `daemon-liveness` (stale lockfile / dead daemon)

**Symptom.** Doctor reports `daemon-liveness: error — pid <N> in lockfile is not running`.
**Cause.** A previous daemon crashed without cleaning up `~/.figma-mcp/daemon.lock`. The next shim still trusts the lockfile and refuses to spawn.
**Fix.** Delete `~/.figma-mcp/daemon.lock` and re-run any tool. The shim auto-spawns a fresh daemon.

**Auto-fix:** `figma-mcp doctor --fix` clears the stale lockfile in this case.

### `plugin-pairing` (plugin not paired)

**Symptom.** Doctor reports `plugin-pairing: warn — no plugin connected on slot 0`.
**Cause.** Figma Desktop is not running, OR the bridge plugin window is closed, OR the plugin failed to connect over loopback.
**Fix.** Open Figma. Run **Plugins → Development → Bridge**. Re-run doctor — the check should flip to `ok`. If it stays `warn`, check the plugin's console (right-click in Figma's plugin window → Show Console) for the WebSocket error.

### `ai-client-configs` (config drift)

**Symptom.** Doctor reports `ai-client-configs: warn — entry path mismatch for cursor`.
**Cause.** A previous setup run wrote a config; later, the user moved/renamed/uninstalled the binary, or hand-edited the config. The entry now points at a stale path.
**Fix.** Re-run `npx @bromso/figma-mcp setup --client cursor` to overwrite. Or hand-edit the JSON. The setup CLI is idempotent.

**Auto-fix:** `figma-mcp doctor --fix` re-applies the per-client setup write against every detected client (idempotent for already-correct ones).

### `recent-errors` (daemon log tail)

**Symptom.** Doctor reports `recent-errors: error — 3 errors in the last 60s`.
**Cause.** The daemon is hitting an unhandled error path — usually a transport disconnect, occasionally a tool-handler crash.
**Fix.** Inspect `~/.figma-mcp/daemon.log` (the last 60s window). If the error is `ECONNRESET`, the bridge plugin disconnected — restart it. Otherwise, file an issue with the log excerpt.

### `oauth` (OAuth token state)

**Symptom.** Doctor reports `oauth: warn — no OAuth token file at ~/.figma-mcp/oauth.json` or `oauth: error — OAuth token file is missing required fields`.

**Cause.** Either the OAuth flow has never been completed (warn), the access token is within 60s of expiry (warn — auto-refreshes on the next REST call), or the file is corrupt/unreadable (error).

**Fix.**
- Warn (no token file) and you only use REST tools → set `FIGMA_API_KEY` (PAT-only is a valid posture).
- Warn (no token file) and you want full OAuth → run `figma-mcp setup --cloud --oauth`.
- Warn (expires within 60s) → no action needed; the daemon refreshes the token lazily.
- Error → delete `~/.figma-mcp/oauth.json` and re-run `figma-mcp setup --cloud --oauth`.

The check is complementary to `figma-api-key` — both can be set up; the daemon prefers the OAuth bearer token when present and falls back to the PAT otherwise.

### `figma-api-key` (REST tools won't work)

**Symptom.** Doctor reports `figma-api-key: warn — FIGMA_API_KEY not set`.

**Cause.** The daemon was started without `FIGMA_API_KEY` in its environment. The 20 tools in the [rest pack](/docs/tools/rest) won't function — they return `E_FIGMA_API_KEY_MISSING` per call.

**Fix.** Set the env var before starting the daemon:

```bash
export FIGMA_API_KEY=figd_…
figma-mcp doctor   # re-check
```

The check is a `warn`, not an `error` — the daemon boots fine without the key; only the REST pack is gated.

### `socket-conflict` (port/socket already taken)

**Symptom.** Doctor reports `socket-conflict: error — /tmp/figma-mcp.sock is in use by pid <N>, not the daemon`.
**Cause.** Another process owns the IPC socket. Could be a leftover from a crashed daemon, or another tool that picked the same path.
**Fix.** Identify the offender (`lsof -nP /tmp/figma-mcp.sock`), kill it, delete the socket file, re-run.

## Specific gotchas (from the design doc)

### Multiple AI clients, one Figma

Only one bridge plugin pairing is active at a time. Multiple AI clients connect to the same daemon, multiplexed over the daemon's single WebSocket slot — so concurrent tool calls from two clients are serialized rather than queued per-client. There is no per-client request isolation in v1.

### Editor-type mismatch

The bridge plugin's manifest declares `["figma", "figjam"]` as supported editor types. If you import it into a Figma Slides file, the plugin runs but tools that depend on `figma.currentPage.type === "PAGE"` may misbehave. Doctor catches this via the daemon's diagnostics report from the plugin side.

### Version drift between shim and daemon

A long-running daemon predates the shim's last `npx` cache update — the shim has the newer code but the daemon is older. Stop the daemon (`pkill -f figma-mcp`); the next shim invocation respawns it at the new version. Doctor reports the version pair under the `daemon-liveness` check.

### Streamable HTTP vs stdio

Streamable HTTP is the cloud-mode entry; stdio is the local-mode entry. Mixing the two for a single AI client is unsupported (it would multiplex two independent sessions). Pick one mode per AI client; `setup` writes the matching entry based on `--cloud` (or its absence).

### Cloud-mode pairing code expiration

Pairing codes minted by `setup --cloud` expire after 5 minutes. If you delay drag-importing the bridge plugin past the TTL, the relay rejects the plugin's pair attempt and returns `PairingExpired`. Re-run `setup --cloud` to mint a fresh code, then drag-import within the TTL window.

### OAuth flow timed out or failed

`figma-mcp setup --cloud --oauth` opens your browser, polls the relay for 5 minutes, and writes tokens to `~/.figma-mcp/oauth.json` on success. Typical failures:

- `E_OAUTH_START_FAILED: 500` — the relay is missing `FIGMA_OAUTH_CLIENT_ID/SECRET`. The relay operator needs to register a Figma OAuth app and set the secrets via `wrangler secret put`. See [Cloud → OAuth setup → Maintainer-side prerequisites](/docs/get-started/cloud).
- `E_OAUTH_TIMEOUT` — you took longer than 5 minutes to authorize. Re-run `setup --cloud --oauth`.
- `E_OAUTH_SESSION_EXPIRED` — the relay's in-flight session timed out. Re-run `setup --cloud --oauth`.
- `E_OAUTH_REFRESH_NOT_CONFIGURED` (at REST-call time) — the access token expired but `FIGMA_OAUTH_CLIENT_ID/SECRET` aren't set on the daemon, so it can't auto-refresh. Either set those env vars on the daemon (matching the relay's secrets) or re-run `setup --cloud --oauth` to mint a fresh token pair.

PAT-based cloud mode (Phase 6) keeps working unchanged — OAuth is opt-in.

## When all else fails

- Re-run `figma-mcp doctor --json` and attach the output to a GitHub issue.
- Include your AI client name + version, OS, and Figma Desktop version.
- The issue template at https://github.com/bromso/bro/issues prompts for the same.
